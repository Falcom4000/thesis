\chapter{Method}
\label{chap:method}

RapidGS accelerates 3DGS training by controlling the trajectory of the Gaussian set, the Gaussian--tile work generated by that set, the backend cost of a training step, and the number of executed steps. This chapter describes the method at the algorithmic level. Implementation details and runtime accounting are described in Chapter~\ref{chap:implementation}.

The method is organized around two principles. First, the retained Gaussian instances use the standard 3DGS representation, projection equations, opacity activation, and front-to-back alpha-blending rule. Compact tile coverage is an explicitly controlled approximation and is specified separately below. Second, every acceleration mechanism should have a clear cost target. Structure control targets redundant Gaussian growth, compact rasterization targets unnecessary Gaussian-tile instances, backend fusion targets optimizer-path overhead, and the short schedule targets repeated step cost.

\section{Overview}

Training alternates between parameter optimization and structure edits. On every iteration, a training view is sampled and rendered. PyTorch computes the RGB loss and its image-space gradient. The fused CUDA backend propagates this gradient to Gaussian parameters, updates Adam moments and parameters, and optionally accumulates densification statistics. At structure-edit clocks, the system uses gradient statistics and multi-view high-error scores to decide which Gaussians should be cloned, split, or pruned.

The implementation executes callbacks in descending priority. Consequently, a scheduled structure callback runs before the ordinary optimization step with the same iteration index. It consumes statistics accumulated by earlier training steps, mutates the Gaussian set, resets or reorders auxiliary state as required, and only then renders the next sampled training view. This ordering matters for reproduction: a callback is not applied after the gradient generated at the identically numbered step.

\caption{RapidGS training loop}
\label{alg:training-loop}
\begin{algorithmic}[1]
\STATE Initialize Gaussian set $\mathcal{G}$ from COLMAP/SfM points
\STATE Initialize CUDA-side parameter tensors, Adam moments, and auxiliary state
\FOR{$t=1$ to $N_{\mathrm{iter}}$}
  \IF{structure-edit clock is active}
    \STATE Estimate multi-view high-error contribution scores
    \STATE Clone or split selected Gaussians
    \STATE Prune low-value or inconsistent Gaussians
    \STATE Synchronize parameters, Adam moments, and auxiliary state
  \ENDIF
  \STATE Sample a training view $(\mathbf{I}_v,\Pi_v)$
  \STATE Render $\hat{\mathbf{I}}_v$ with compact tile-instance generation
  \STATE Compute $\mathcal{L}_{\mathrm{rgb}}$ in PyTorch
  \STATE Pass $\partial\mathcal{L}_{\mathrm{rgb}}/\partial\hat{\mathbf{I}}_v$ to the fused CUDA backend
  \STATE Run CUDA backward, Adam update, and optional densification-statistics accumulation
\ENDFOR
\end{algorithmic}
\end{algorithm}

\section{Design Rationale}

A fast 3DGS trainer must avoid two opposite failure modes. If it grows too few Gaussians, the model lacks capacity and loses detail. If it grows too many Gaussians, training becomes expensive and later pruning has to spend effort undoing earlier decisions. The standard view-space gradient signal is useful, but it is local and view-dependent. It can identify where a projected Gaussian is sensitive to image-space movement, but it cannot by itself tell whether the signal corresponds to stable multi-view structure or a transient residual.

The proposed design keeps the gradient signal because it is inexpensive and directly tied to the differentiable rasterizer. It then adds a multi-view contribution gate to determine whether the Gaussian actually participates in explaining high-error pixels across sampled views. The combination is more selective than either signal alone. A high gradient without multi-view contribution can indicate local instability. Multi-view high-error contribution without a sufficient gradient may indicate residual error nearby, but not necessarily a useful clone or split candidate. The evidence for growth is strongest when both signals agree.

Pruning uses related observations with a different interpretation. A low-opacity Gaussian is unlikely to contribute much color. A large or abnormal Gaussian can explain too broad a region and create unnecessary tile work. A split parent can become redundant after its children are inserted. A Gaussian that repeatedly participates in high-error regions late in training may be part of an inconsistent explanation rather than useful structure. These deletion rules are intentionally not the mirror image of densification rules; adding and deleting have different risks and should run on different clocks.

\section{Cost-Aware Structure Editing}

Densification and pruning are both discrete edits to the Gaussian set. An edit $a$ transforms the current set $\mathcal{G}$ into $\mathcal{G}_a$ and changes both reconstruction loss and training cost:
\begin{equation}
\Delta\mathcal{L}(a)
=
\mathcal{L}(\mathcal{G}_a)-\mathcal{L}(\mathcal{G}),
\quad
\Delta\mathcal{C}(a)
=
\mathcal{C}(\mathcal{G}_a)-\mathcal{C}(\mathcal{G}).
\end{equation}
An ideal edit would satisfy
\begin{equation}
\Delta\mathcal{L}(a)+\lambda\Delta\mathcal{C}(a)<0.
\end{equation}
This condition is not directly evaluated in the implementation because testing every candidate edit would require re-optimization. Instead, RapidGS uses low-cost proxy signals that approximate where additional capacity is useful and where existing capacity has low value.

The edit view also explains why densification and pruning should not be treated as independent heuristics. A clone or split operation increases the future cost of every following iteration in which the new Gaussian remains active. A pruning operation decreases future cost, but it can raise reconstruction loss if it deletes a useful primitive. The useful question is therefore not whether a Gaussian has a large local signal in isolation. The useful question is whether the expected quality benefit of keeping or adding capacity justifies the repeated runtime cost.

In practice, the method uses action-specific clocks. Growth is most useful while the representation is still discovering scene structure. Later in training, aggressive growth has less time to pay back its cost. Pruning is useful throughout training but becomes especially important after the structure has mostly formed. Delaying stronger pruning also reduces the chance of deleting a Gaussian that would have become useful after nearby primitives move or split.

This leads to an asymmetric policy. Adding a Gaussian and deleting a Gaussian are not inverse operations, because their failure costs are different. A bad addition mostly increases future computation and can later be removed. A bad deletion removes parameters, optimizer moments, and accumulated statistics, and the system can only recover indirectly by growing new Gaussians in nearby regions. RapidGS therefore uses frequent but gated growth, conservative pruning, and late confirmation for more ambiguous deletion signals.

The action-specific view can be written as
\begin{equation}
\mathcal{A}_t
=
\mathcal{A}^{+}\mathbb{1}[t\in\mathcal{T}_{+}]
\cup
\mathcal{A}^{-}\mathbb{1}[t\in\mathcal{T}_{-}]
\cup
\mathcal{A}^{\mathrm{reset}}\mathbb{1}[t\in\mathcal{T}_{\mathrm{reset}}],
\end{equation}
where $\mathcal{T}_{+}$, $\mathcal{T}_{-}$, and $\mathcal{T}_{\mathrm{reset}}$ are the active clocks for growth, pruning, and opacity reset. The clocks are part of the method rather than an implementation accident. They determine how long a new Gaussian can repay its cost, how much evidence is required before deletion, and when opacity saturation should be relaxed.

\section{Multi-View High-Error Contribution Score}

The standard 3DGS densification rule relies heavily on view-space gradient magnitude. This signal is useful because it indicates local optimization pressure, but it is also sensitive to single-view noise, occlusion boundaries, background competition, and transient high-error pixels. RapidGS keeps the gradient signal but requires selected candidates to pass a multi-view contribution gate.

Given a sampled set of score views $\mathcal{V}_K$, a high-error mask $M_v(\mathbf{p})$ is computed from the RGB residual in each view. A metric-count render then counts whether Gaussian $i$ passes the renderer's per-fragment alpha test at a high-error pixel before transmittance termination:
\begin{equation}
r_i^{\mathrm{mv}}
=
\sum_{v\in\mathcal{V}_K}
\sum_{\mathbf{p}}
M_v(\mathbf{p})A_{i,v}(\mathbf{p}),
\end{equation}
where the implemented indicator is
\begin{equation}
A_{i,v}(\mathbf{p})
=
\mathbb{1}\!\left[
i\ \text{is visited before early termination at }\mathbf{p}
\right]
\cdot
\mathbb{1}\!\left[
\alpha_{i,v}(\mathbf{p})\ge\tau_\alpha
\right].
\end{equation}
The implementation uses $\tau_\alpha=1/255$ and terminates a pixel when its running transmittance falls below $\tau_T=10^{-4}$. Gaussians after this termination point are not counted. The count is binary per Gaussian--pixel visit: it does not use the raw opacity alone, but it also does not weight the count by $T_{i,v}(\mathbf{p})\alpha_{i,v}(\mathbf{p})$. Transmittance affects the statistic through visibility and early termination. This exact definition avoids conflating metric count with either bounding-box overlap or a continuous contribution integral.

The distinction between contribution count and bounding-box overlap is central. A projected bounding box only indicates that a Gaussian might affect a tile or pixel. It ignores opacity, Gaussian falloff, transmittance, and early termination in alpha blending. A contribution-aware count is closer to the training computation because it follows the same visibility path that produces the rendered RGB image. This makes the score better aligned with the question asked by density control: whether a Gaussian is actually involved in image formation where the model currently has residual error.

The score is deliberately simple. It counts effective participation rather than estimating the exact loss decrease from a hypothetical edit. Exact edit evaluation would require inserting or deleting a candidate, running additional optimization, and measuring multi-view loss again. That would be too expensive for a low-frequency callback. A metric-count render provides a cheaper proxy that can be computed on a small sampled set of views.

The high-error mask is computed from a score render with no final clamp to $[0,1]$. The SH color entering blending is still clamped component-wise to be non-negative, matching the training rasterizer. For a score view $v$,
\begin{equation}
e_v(\mathbf{p})
=
\frac{1}{3}
\left\|
\hat{\mathbf{I}}_v(\mathbf{p})-\mathbf{I}_v(\mathbf{p})
\right\|_1,
\end{equation}
and the normalized mask is
\begin{equation}
M_v(\mathbf{p})
=
\mathbb{1}
\left[
\frac{e_v(\mathbf{p})-\min_{\mathbf{q}} e_v(\mathbf{q})}
{\max_{\mathbf{q}} e_v(\mathbf{q})-\min_{\mathbf{q}} e_v(\mathbf{q})+\epsilon}
>
\tau_e
\right].
\end{equation}
The implementation sets $\epsilon$ implicitly by branching: if the range is at most $10^{-12}$, the normalized map is set to zero; otherwise ordinary min--max normalization is used. The threshold is $\tau_e=0.1$. Min--max normalization is sensitive to a single extreme residual, so this is an implementation choice rather than a robust-statistics claim. All compared configurations use the same rule. A percentile normalization would change the candidate set and must be evaluated as a separate method variant.

At each callback, views are sampled uniformly without replacement from a cached list of training views using Python's seeded pseudo-random generator. The main configuration uses
\begin{equation}
K=\min(5,|\mathcal{V}_{\mathrm{train}}|).
\end{equation}
Score rendering is not an every-step operation. During densification it runs every 100 iterations from iteration 600 through the callback window ending at 15,000. The late pruning callback runs every 3,000 iterations from 18,000 to the 23,000-step training boundary. For each sampled view, the current code performs one forward render to construct the residual map and a second metric-count render using the resulting Boolean mask. Both renders use the same active SH degree, background color, compact-box multiplier, alpha cutoff, and early-termination rule as the maintained rasterizer. If a ground-truth alpha channel exists, the target RGB is composited with the same background before residual and view-loss computation.

For densification, the per-Gaussian score is averaged over the sampled score views:
\begin{equation}
s_i^{+}
=
\left\lfloor
\frac{1}{|\mathcal{V}_K|}
\sum_{v\in\mathcal{V}_K}
\sum_{\mathbf{p}}M_v(\mathbf{p})A_{i,v}(\mathbf{p})
\right\rfloor .
\end{equation}
For pruning, the same contribution evidence is interpreted later in training and can be weighted by view-level residual:
\begin{equation}
\tilde{s}_i^{-}
=
\sum_{v\in\mathcal{V}_K}
\ell_v
\sum_{\mathbf{p}}M_v(\mathbf{p})A_{i,v}(\mathbf{p}),
\quad
s_i^{-}=\operatorname{norm}(\tilde{s}_i^{-}).
\end{equation}
Here $\ell_v=0.8\mathcal{L}_{1,v}+0.2\mathcal{L}_{\mathrm{DSSIM},v}$ and $\operatorname{norm}$ is min--max normalization over the current Gaussian set, with the same $10^{-12}$ constant-range guard. The integer floor in $s_i^{+}$ is part of the implementation. Since the gate uses the strict comparison $s_i^{+}>5$, a Gaussian must have an integer average metric count of at least six in the main configuration. The two scores share the same rendering statistic but answer different questions. The growth score asks where extra capacity may reduce residual error. The pruning score asks whether an existing primitive repeatedly appears in high-error regions late enough that it may be part of an inconsistent explanation.

\section{Multi-View Consistent Densification}

RapidGS uses multi-view consistent densification (VCD) for score-gated growth.
The name follows FastGS; the equations below specify this implementation's
score, threshold, callback schedule, and gradient statistics.

Growth candidates are first selected by view-space gradient statistics accumulated during training. They are then filtered by the multi-view high-error contribution score. Small Gaussians are cloned to add local capacity, while large Gaussians are split into smaller children to refine coarse structure. The intent is to allocate new primitives to regions where local gradient pressure and multi-view residual evidence agree.

Clone and split serve different structural roles. Cloning preserves the scale of a small Gaussian and adds nearby capacity so that local appearance or geometry can be represented with more degrees of freedom. Splitting is used when a Gaussian is too large and should be replaced by smaller children. This distinction follows the standard 3DGS intuition, but VCD makes the candidate set more selective by applying the multi-view score criterion.

VCD also limits a common source of runtime growth. If a single-view gradient spike creates many candidates, unconstrained densification can increase the Gaussian count even when the residual is caused by occlusion ordering, background mismatch, or temporary color competition. Requiring multi-view evidence reduces this type of redundant growth. It does not guarantee that every edit is optimal, but it makes the edit policy more aligned with stable scene structure.

Let $\mathbf{h}_{i,t}$ be the projected-mean gradient accumulated over all contributing pixels of training step $t$. The fused backward path stores three quantities over one densification interval:
\begin{equation}
d_i=\sum_t \mathbb{1}[i\text{ is visible at }t],
\quad
g_i^{\mathrm{sgn}}=\sum_t\|\mathbf{h}_{i,t}\|_2,
\quad
g_i^{\mathrm{abs}}=\sum_t\left\|
\sum_{\mathbf{p}}|\mathbf{h}_{i,t}(\mathbf{p})|
\right\|_2 .
\end{equation}
The first gradient is cancellation-sensitive within a step because pixel gradients are summed before taking the norm. The absolute variant accumulates component-wise absolute pixel gradients before taking the norm and is therefore more sensitive to large Gaussians whose opposing pixel gradients would otherwise cancel. Both are converted to normalized-device-coordinate scale before accumulation. The base clone and split candidates are
\begin{equation}
c_i^0=
\mathbb{1}[g_i^{\mathrm{sgn}}\ge \tau_g d_i],
\quad
p_i^0=
\mathbb{1}[g_i^{\mathrm{abs}}\ge \tau_a d_i].
\end{equation}
The multi-view gate is then applied as
\begin{equation}
c_i=
c_i^0\land\mathbb{1}[s_i^{+}>\tau_{\mathrm{imp}}],
\quad
p_i=
p_i^0\land\mathbb{1}[s_i^{+}>\tau_{\mathrm{imp}}].
\end{equation}
The final action also depends on scale:
\begin{equation}
\mathcal{C}_i=c_i\land\mathrm{small}_i,
\quad
\mathcal{P}_i=p_i\land\neg\mathrm{small}_i.
\end{equation}
This separates the evidence for growth from the geometry of the operation. The score and gradient decide whether extra capacity is useful; the scale test decides whether the capacity should preserve the current footprint or refine it.

The implemented size test is
\begin{equation}
\mathrm{small}_i
=
\mathbb{1}\!\left[
\max_k \exp(s_{i,k})
\le
\rho R_{\mathrm{cam}}
\right],
\end{equation}
where $\rho=0.001$ and $R_{\mathrm{cam}}$ is 1.1 times the maximum distance of a training camera center from the mean camera center. A clone is an exact copy of the selected small parent at insertion time: mean, SH coefficients, opacity logit, scale, and rotation are copied, while all child Adam moments and the VCP counter are initialized to zero. The two identical parameter rows can subsequently diverge because rasterization gradients and Adam histories evolve independently.

For a split parent, exactly two children are sampled from the parent ellipsoid and use a reduced scale:
\begin{equation}
\boldsymbol{\mu}'_{i,k}
=
\boldsymbol{\mu}_i+
R(\mathbf{q}_i)\operatorname{diag}(\exp(\mathbf{s}_i))\boldsymbol{\epsilon}_{i,k},
\quad
\boldsymbol{\epsilon}_{i,k}\sim\mathcal{N}(\mathbf{0},I),
\quad
\mathbf{s}'_{i,k}=\mathbf{s}_i+\log(0.625),
\quad k\in\{1,2\}.
\end{equation}
The children copy the parent's SH coefficients, opacity logit, and raw rotation, start with zero Adam moments, and are appended before the parent is removed. This replacement behavior prevents the coarse parent and its children from explaining the same region indefinitely.

\begin{algorithm}[H]
\caption{Implemented densification callback}
\label{alg:structure-edit}
\begin{algorithmic}[1]
\STATE Read normalized signed and absolute view-space gradient statistics
\STATE Render score views and compute high-error masks
\STATE Run metric-count render to obtain $r_i^{\mathrm{mv}}$
\STATE Select clone/split candidates using gradient, importance score, and scale tests
\STATE Append clone children and split children to the Gaussian tensors
\STATE Append zero Adam moments and VCP counters for all new children
\STATE Remove split parents with one aligned mask
\STATE Build ordinary prune mask from opacity and, after iteration 3,000, world scale
\STATE Stochastically restrict that mask using the current pruning score
\STATE Compact all aligned state, cap opacity at 0.8, and clear opacity moments
\STATE Reset the three-channel densification statistics for the next interval
\end{algorithmic}
\end{algorithm}

\section{Pruning}

Pruning removes Gaussians that are unlikely to justify their future training cost. The current implementation has three distinct deletion paths: split-parent replacement, ordinary pruning inside a densification callback, and late VCP. Keeping these paths separate is important because their candidate conditions and scheduling are different.

Split-parent removal is deterministic and immediate. Ordinary pruning first marks activated opacity below 0.005. After iteration 3,000, it additionally marks a Gaussian when its maximum world-space scale exceeds $0.1R_{\mathrm{cam}}$. The pruning score computed during the same callback does not independently add candidates. Instead, the implementation samples at most
\begin{equation}
B_{\mathrm{ord}}=\left\lfloor 0.5
\left|P_{\mathrm{opacity}}\cup P_{\mathrm{scale}}\right|
\right\rfloor
\end{equation}
indices without replacement using weights
\begin{equation}
w_i=\frac{1}{10^{-6}+1-s_i^{-}},
\end{equation}
and intersects the sampled set with the ordinary opacity/scale mask. Thus high pruning score makes an otherwise ordinary prune candidate more likely to survive the stochastic restriction, but it is not itself sufficient for deletion in this path. This score-conditioned subsampling is separate from late VCP, where $s_i^{-}>\tau_p$ is an explicit candidate condition.

\subsection{View-Consistency Pruning}

VCP denotes \emph{view-consistency pruning}. It is a later-stage consistency check, not ordinary opacity pruning and not the inverse of the densification gate. Its inputs are the activated opacity $o_i$, normalized multi-view pruning score $s_i^{-}$, persistent confirmation counter $H_i$, confirmation requirement $h$, and per-callback budget fraction $\beta$. The candidate condition is
\begin{equation}
c_i^{-}
=
\mathbb{1}[o_i<\tau_o]
\lor
\mathbb{1}[s_i^{-}>\tau_p],
\end{equation}
with $\tau_o=0.1$ and $\tau_p=0.9$ in the main configuration. The counter is updated only at VCP callbacks:
\begin{equation}
H_i\leftarrow
\begin{cases}
\min(H_i+1,h), & c_i^{-}=1,\\
0, & c_i^{-}=0.
\end{cases}
\end{equation}
A Gaussian is confirmed when $H_i\ge h$. If $0<\beta<1$, at most $\lceil\beta N_c\rceil$ confirmed candidates are deleted, where $N_c$ is the number of confirmed candidates. Candidates are ordered by descending $s_i^{-}$; low-opacity candidates are assigned priority at least one, so they are not placed behind lower-priority score-only candidates. The main configuration uses $h=1$ and $\beta=1$, while the implementation exposes both controls for delayed and budgeted variants.

The deletion order is also explicit. A densification callback first appends clone and split children, then removes split parents, then applies the ordinary opacity/scale mask produced by adaptive density control. A VCP callback is separate and late: it combines low opacity and VCP score, applies confirmation and budget, and then compacts the state. Opacity reset is not a deletion operation. Scale pruning belongs to adaptive density control, while split-parent replacement has priority because it is executed immediately after child insertion.

Across the complete training procedure, the possible deletion causes can be summarized as
\begin{equation}
m_i^{\mathrm{prune}}
=
m_i^{\mathrm{opacity}}
\lor
m_i^{\mathrm{scale}}
\lor
m_i^{\mathrm{score}}
\lor
m_i^{\mathrm{replace}}.
\end{equation}
The terms have different meanings and are not evaluated in one monolithic mask. The opacity and scale terms belong to densification callbacks, the score term belongs to late VCP, and the replacement term is executed immediately after split-child insertion.

\subsection{Opacity Control}

Opacity is also edited without changing Gaussian count. At opacity-reset callbacks, raw opacity logits are capped at $\operatorname{logit}(0.01)$ and all opacity Adam moments are cleared. After each densification callback, opacity logits are capped at $\operatorname{logit}(0.8)$ and the opacity moments are again cleared. These operations prevent highly saturated opacity and stale momentum from immediately undoing a structural reset. For the main black-background benchmark, the extra reset at iteration 500 is inactive; it is triggered only when the dataset background is non-black.

\section{Short Schedule Under a Quality Constraint}

RapidGS uses a shorter training schedule because controlled structure editing and a lower-cost backend make the model reach the target quality earlier. The schedule is not chosen as an independent speed trick. If the Gaussian set is unstable, shortening the run can reduce quality. If pruning is too aggressive, a short schedule gives the model less time to recover. The schedule therefore belongs to the same system design as densification, pruning, compact rasterization, and fused optimization.

The schedule criterion is multi-metric. A setting should not be accepted only because it has good PSNR or only because it has low LPIPS. The selected schedule should preserve the reference quality region across PSNR, SSIM, and LPIPS while reducing total optimization time. This is why the experiments report all three metrics and why further schedule reductions are treated as boundary cases until they are validated by full runs.

\section{Fused Training Backend}

The fused backend keeps the RGB loss in PyTorch but avoids exposing all per-Gaussian gradients to the PyTorch optimizer. The only quantity that must cross this boundary after the forward pass is
\begin{equation}
\frac{\partial\mathcal{L}_{\mathrm{rgb}}}{\partial\hat{\mathbf{I}}_v}.
\end{equation}
CUDA kernels propagate this image-space gradient through alpha blending, 2D conic parameters, projected means, 3D covariance, opacity, SH color, and 3D means. The backend then updates Adam moments and Gaussian parameters directly. This design reduces \texttt{.grad} tensor traffic, general optimizer bookkeeping, and the kernel-launch overhead associated with separate optimizer operations.

This design keeps the loss interface flexible while specializing the expensive Gaussian path. Loss composition remains in Python, so the training loop can still combine RGB terms, logging, and evaluation in a familiar way. The backend only requires the gradient of the scalar loss with respect to the rendered image. From that point onward, all per-Gaussian work is handled in a layout that knows the Gaussian structure and can update parameters and moments consistently.

The fused backend is especially useful because the Gaussian set is dynamic. When new Gaussians are appended, Adam moments must be appended with matching rows. When Gaussians are pruned, the corresponding moments must be removed. When Gaussians are reordered, moments and auxiliary state must be reordered exactly the same way. Keeping this state in a single backend reduces the risk of state mismatch.

\section{Compact Rasterization}

The rasterizer first assigns Gaussians to screen-space tiles. A bounding-box assignment can generate many Gaussian-tile pairs whose actual alpha contribution is negligible, especially near the corners of the projected footprint. RapidGS uses a compact contribution test during preprocessing. For each candidate tile, the rasterizer estimates whether the Gaussian can reach an effective alpha contribution anywhere in that tile. Only passing Gaussian-tile pairs are emitted into the instance list used by sorting, blending, and backward execution.

For Gaussian $i$, let the screen-space exponent cost be
\begin{equation}
q_i(\mathbf{p})
=
\frac{1}{2}
(\mathbf{p}-\boldsymbol{\mu}_{i}^{2D})^\top
Q_i
(\mathbf{p}-\boldsymbol{\mu}_{i}^{2D}),
\qquad
\alpha_i(\mathbf{p})=o_i\exp[-q_i(\mathbf{p})],
\end{equation}
where $Q_i$ is the positive-definite conic matrix and $o_i$ is the activated opacity. For a tile region $\mathcal{R}$, the implementation finds the point in the tile that minimizes this quadratic form using the geometry of the projected ellipse and tile rectangle. This gives
\begin{equation}
q_{i,\mathcal{R}}^{\min}
=
\min_{\mathbf{p}\in\mathcal{R}}q_i(\mathbf{p}),
\qquad
\alpha_{i,\mathcal{R}}^{\max}
=
o_i\exp[-q_{i,\mathcal{R}}^{\min}].
\end{equation}
The Gaussian--tile pair is retained when
\begin{equation}
q_{i,\mathcal{R}}^{\min}
\le
m\log\!\left(\frac{o_i}{\tau_\alpha}\right),
\end{equation}
where $\tau_\alpha=1/255$ and $m$ is the compact-box multiplier. With $m=1$, the test retains every tile whose maximum possible alpha reaches the renderer cutoff, up to floating-point and rectangle-boundary conventions. The main speed configuration uses $m=0.75$, which is stricter and therefore approximates standard tile coverage. It preserves the standard alpha-blending rule and depth order for retained instances while omitting weak tile instances before sorting and blending. The approximation error is controlled by the selected alpha threshold and multiplier; this does not imply bitwise equivalence to bounding-box-only rasterization.

The value $m=0.75$ was selected by the compact-box ablation reported in Chapter~\ref{chap:experiments}. Relative to the previous $m=1$ operating point, it reduced the nine-scene mean training time while preserving the reported PSNR, SSIM, and LPIPS region. This empirical quality check is necessary because $m<1$ no longer provides the strict cutoff-equivalence guarantee available at $m=1$.

The same tile predicate is evaluated twice in the forward setup. Preprocessing first counts the exact number of retained tiles for each visible Gaussian, enabling a correctly sized instance buffer and prefix-sum offsets. Instance generation then repeats the predicate and writes exactly those Gaussian--tile pairs. Candidate rectangles with at most four tiles are enumerated by the owning thread; larger rectangles are distributed cooperatively across a warp. These scheduling choices do not alter the predicate, but they avoid assigning a long tile loop to one thread for a large projected Gaussian.

Compact rasterization complements structure control. Structure control reduces the number of primitives that survive in the model. Compact rasterization reduces the amount of work generated by the primitives that remain. These are different levers: a scene can have a moderate number of Gaussians but many tile instances if the Gaussians are large, and a scene can have many small Gaussians whose tile lists remain short. The training system benefits from controlling both quantities.

\section{Putting the Components Together}

The components are designed to reinforce each other. VCD limits early redundant growth, which reduces future rasterization and optimizer work. Pruning removes low-value or inconsistent capacity, which lowers later step cost. Compact rasterization reduces the work of retained Gaussians by avoiding negligible tile instances. The fused backend lowers the cost of gradient propagation and Adam updates for the active Gaussian set. A shorter schedule then reduces the number of times the remaining step cost is paid.

This combined interpretation is important for evaluating the method. A single ablation may not capture the full effect because the components change the operating point of one another. For example, compact rasterization can make a larger Gaussian set more affordable, while better structure control can make a short schedule safer. The final system should therefore be judged by both end-to-end training time and intermediate cost indicators such as Gaussian count, tile-instance count, runtime breakdown, and reconstruction quality.
